## [ ] Shared memory write-conflict strategy

**What:** Design and implement optimistic locking for `shared_memory` writes. Check `updated_at` before writing; retry or merge if stale.

**Why:** When multiple agents write to shared_memory concurrently (asyncio.gather), last-write-wins. In a 4-agent discussion where two agents update the same memory category simultaneously, one update silently disappears.

**Pros:** Correct behavior under concurrent writes.

**Cons:** Adds complexity to MemoryService.

**Context:** Low probability in Phases 1-2 (agents rarely target the same memory category simultaneously). Becomes real in Phase 3 when all 4 agents are active in a discussion. Design the solution before Phase 3, implement when first conflict is observed.

**How to apply:** Add `updated_at` check to `MemoryService.save()`:
```python
# Optimistic lock: only write if memory hasn't changed since we read it
UPDATE shared_memory SET content=$1, updated_at=NOW()
WHERE id=$2 AND updated_at=$3
RETURNING id;
# If no rows returned → conflict → retry or merge
```

**Depends on:** Phase 3 multi-agent discussion implementation.
